# 路线图模板

用于 `.planning/ROADMAP.md` 的模板。

## 初始路线图（v1.0 绿地）

```markdown
# 路线图：[项目名称]

## 概述

[一段描述从开始到完成的旅程]

## 阶段

**阶段编号：**
- 整数阶段（1、2、3）：计划的里程碑工作
- 小数阶段（2.1、2.2）：紧急插入（标记为 INSERTED）

小数阶段按数字顺序出现在其周围的整数之间。

- [ ] **阶段 1：[名称]** - [一句话描述]
- [ ] **阶段 2：[名称]** - [一句话描述]
- [ ] **阶段 3：[名称]** - [一句话描述]
- [ ] **阶段 4：[名称]** - [一句话描述]

## 阶段详情

### 阶段 1：[名称]
**目标**：[此阶段交付的内容]
**依赖**：无（第一阶段）
**需求**：[REQ-01、REQ-02、REQ-03]
**成功标准**（必须为真）：
  1. [用户视角的可观察行为]
  2. [用户视角的可观察行为]
  3. [用户视角的可观察行为]
**计划**：[计划数量，例如"3 个计划"或"TBD"]

计划：
- [ ] 01-01：[第一个计划的简要描述]
- [ ] 01-02：[第二个计划的简要描述]
- [ ] 01-03：[第三个计划的简要描述]

### 阶段 2：[名称]
**目标**：[此阶段交付的内容]
**依赖**：阶段 1
**需求**：[REQ-04、REQ-05]
**成功标准**（必须为真）：
  1. [用户视角的可观察行为]
  2. [用户视角的可观察行为]
**计划**：[计划数量]

计划：
- [ ] 02-01：[简要描述]
- [ ] 02-02：[简要描述]

### 阶段 2.1：关键修复（INSERTED）
**目标**：[阶段之间插入的紧急工作]
**依赖**：阶段 2
**成功标准**（必须为真）：
  1. [修复实现的内容]
**计划**：1 个计划

计划：
- [ ] 02.1-01：[描述]

### 阶段 3：[名称]
**目标**：[此阶段交付的内容]
**依赖**：阶段 2
**需求**：[REQ-06、REQ-07、REQ-08]
**成功标准**（必须为真）：
  1. [用户视角的可观察行为]
  2. [用户视角的可观察行为]
  3. [用户视角的可观察行为]
**计划**：[计划数量]

计划：
- [ ] 03-01：[简要描述]
- [ ] 03-02：[简要描述]

### 阶段 4：[名称]
**目标**：[此阶段交付的内容]
**依赖**：阶段 3
**需求**：[REQ-09、REQ-10]
**成功标准**（必须为真）：
  1. [用户视角的可观察行为]
  2. [用户视角的可观察行为]
**计划**：[计划数量]

计划：
- [ ] 04-01：[简要描述]

## 进度

**执行顺序：**
阶段按数字顺序执行：2 → 2.1 → 2.2 → 3 → 3.1 → 4

| 阶段 | 已完成计划 | 状态 | 完成时间 |
|-------|----------------|--------|-----------|
| 1. [名称] | 0/3 | 未开始 | - |
| 2. [名称] | 0/2 | 未开始 | - |
| 3. [名称] | 0/2 | 未开始 | - |
| 4. [名称] | 0/1 | 未开始 | - |
```

<guidelines>
**初始规划（v1.0）：**
- 阶段数量取决于深度设置（快速：3-5，标准：5-8，综合：8-12）
- 每个阶段交付连贯的内容
- 阶段可以有 1+ 个计划（如果 >3 个任务或多个子系统则拆分）
- 计划使用命名：{phase}-{plan}-PLAN.md（例如，01-02-PLAN.md）
- 无时间估计（这不是企业项目管理）
- 进度表由执行工作流更新
- 计划数量最初可以是"TBD"，在规划期间完善

**成功标准：**
- 每个阶段 2-5 个可观察行为（从用户视角）
- 在路线图创建期间与需求交叉检查
- 流向下游到 plan-phase 中的 `must_haves`
- 执行后由 verify-phase 验证
- 格式："用户可以[操作]"或"[东西]工作/存在"

**里程碑发布后：**
- 在 `<details>` 标签中折叠完成的里程碑
- 为即将到来的工作添加新的里程碑部分
- 保持连续的阶段编号（永远不要从 01 重新开始）
</guidelines>

<status_values>
- `未开始` - 尚未开始
- `进行中` - 正在工作
- `完成` - 完成（添加完成日期）
- `推迟` - 推迟到以后（带有原因）
</status_values>

## 里程碑分组路线图（v1.0 发布后）

完成第一个里程碑后，使用里程碑分组重新组织：

```markdown
# 路线图：[项目名称]

## 里程碑

- ✅ **v1.0 MVP** - 阶段 1-4（已于 YYYY-MM-DD 发布）
- 🚧 **v1.1 [名称]** - 阶段 5-6（进行中）
- 📋 **v2.0 [名称]** - 阶段 7-10（已规划）

## 阶段

<details>
<summary>✅ v1.0 MVP（阶段 1-4）- 已于 YYYY-MM-DD 发布</summary>

### 阶段 1：[名称]
**目标**：[此阶段交付的内容]
**计划**：3 个计划

计划：
- [x] 01-01：[简要描述]
- [x] 01-02：[简要描述]
- [x] 01-03：[简要描述]

[... 剩余的 v1.0 阶段 ...]

</details>

### 🚧 v1.1 [名称]（进行中）

**里程碑目标：** [v1.1 交付的内容]

#### 阶段 5：[名称]
**目标**：[此阶段交付的内容]
**依赖**：阶段 4
**计划**：2 个计划

计划：
- [ ] 05-01：[简要描述]
- [ ] 05-02：[简要描述]

[... 剩余的 v1.1 阶段 ...]

### 📋 v2.0 [名称]（已规划）

**里程碑目标：** [v2.0 交付的内容]

[... v2.0 阶段 ...]

## 进度

| 阶段 | 里程碑 | 已完成计划 | 状态 | 完成时间 |
|-------|-----------|----------------|--------|-----------|
| 1. 基础 | v1.0 | 3/3 | 完成 | YYYY-MM-DD |
| 2. 功能 | v1.0 | 2/2 | 完成 | YYYY-MM-DD |
| 5. 安全 | v1.1 | 0/2 | 未开始 | - |
```

**注意事项：**
- 里程碑表情符号：✅ 已发布、🚧 进行中、📋 已规划
- 完成的里程碑在 `<details>` 中折叠以提高可读性
- 当前/未来里程碑展开
- 连续阶段编号（01-99）
- 进度表包括里程碑列
